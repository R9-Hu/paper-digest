    HarnessX: A Composable, Adaptive, and Evolvable Agent Harness Foundry¶

    🕒 Published (v1): 2026-06-12 00:00 UTC · Source: HuggingFace · link

    Why this paper was selected

    HarnessX: composable, adaptive, evolvable agent harness foundry for LLM agents

    TL;DR¶

    HarnessX is a "harness foundry" that treats the agent runtime scaffold (prompts, tools, memory, control flow) as a first-class typed object that can be composed, automatically evolved via trace-driven multi-agent optimization (AEGIS), and co-trained with the underlying model. Across five benchmarks and three model families it achieves an average absolute improvement of +14.5% (up to +44.0%) over static baselines, demonstrating that harness evolution is a complementary lever to model scaling.

    Problem¶

    Agent harnesses are hand-crafted, monolithic, and static: every model or task change requires bespoke rewriting, components are architecturally entangled so edits break unrelated behavior, and trajectory data collected during execution is discarded rather than recycled into harness or model improvement. No principled framework exists for in-loop, experience-driven harness optimization.

    Method¶

    HarnessX operates on three interlocking layers.

    Harness Composition. The harness is formalized as \(H = (M, C)\) where \(M\) is a model configuration and \(C = (P, S)\) consists of \(P: \text{Hook} \to \text{List[Processor]}\) (processors indexed by eight lifecycle hooks: task_start, step_start, before_model, after_model, before_tool, after_tool, step_end, task_end) and \(S\) (singleton slot resources: tool registry, tracer, workspace, etc.). Every behavior is implemented as a typed Processor with interface async def process(event) -> AsyncIterator[Event], restricted to five outcomes (pass-through, transform, split, intercept, interrupt). A nine-dimensional taxonomy covers: model selection (D1), context assembly (D2), memory management (D3), tool ecosystem (D4), execution environment (D5), evaluation/reward (D6), control/safety (D7), observability (D8), and training bridge (D9). Composition uses a substitution algebra with _singleton_group, _order, and _after metadata fields to enforce mutual-exclusion and ordering without breaking type contracts.

    Harness Adaptation via AEGIS. Harness evolution is cast as an MDP over symbolic artifacts (the operational mirror, Table 2): configurations \(H_t\) are states, typed code-level edits \(e \in E\) are actions, and execution traces \(\Delta T_t\) plus verifier scores \(r_t\) are feedback. The symbolic state is \(s_t = (H_t, T_t)\). A deterministic acceptance gate enforces a seesaw constraint: a candidate \(\tilde{H}_t\) is committed only if it does not regress any previously solved task in \(T_t\). AEGIS realizes the harness-update policy \(\pi_\text{evo}\) as a four-stage pipeline driven by a single meta-agent LLM: Digester compresses per-task traces into summaries; Planner generates an adaptation landscape and exploration guard; Evolver produces candidate harness edits (new processor code, modified prompts, reconfigured tool registries); Critic applies exploit detection and the seesaw constraint. This maps RL pathologies to concrete defenses: reward hacking → Critic exploit detection; catastrophic forgetting → seesaw constraint; under-exploration → Planner exploration guard. On heterogeneous benchmarks (GAIA), variant isolation via ensemble routing maintains separate harness variants per task cluster to prevent cross-task interference.

    Harness-Model Co-Evolution. Traces annotated with verifier rewards from harness evolution populate a mixed-policy replay buffer. Cross-harness GRPO trains the model over trajectories from successive harness versions \(H_1, \ldots, H_t\), so the model internalizes strategies the harness discovered and breaks the "scaffolding ceiling" where further harness edits yield diminishing returns.

    Key Contributions¶

    • Typed harness composition substrate: processors with hook-indexed attachment, singleton-group mutual exclusion, and a substitution algebra making every behavioral change explicit and type-safe.
    • AEGIS: formal operational mirror between symbolic harness evolution and RL (Table 2), enabling RL pathology analysis to translate directly into architectural defenses.
    • Variant isolation strategy that routes heterogeneous tasks to separate harness variants, resolving stagnation on mixed benchmarks.
    • Harness–model co-evolution via cross-harness GRPO over a shared mixed-policy replay buffer, closing the loop between harness adaptation and model training.
    • Empirical validation across five benchmarks, three model families, and up to 15 evolution rounds showing average +14.5% absolute gain.

    Results¶

    • Average absolute gain: +14.5% across 15 model–benchmark configurations (14 of 15 improved).
    • ALFWorld: +44.0% for Qwen3.5-9B (weakest agent, largest gain); +11.2% for Claude Sonnet 4.6.
    • τ³-Bench: +1.1% (near-ceiling baseline, smallest gain among improving configs).
    • GAIA heterogeneous setting: single-harness evolution stagnates; variant-isolation ablation achieves +13.6% without degradation over 15 rounds.
    • Co-evolution: additional +4.7% over harness-only evolution (§6.5).
    • Inverse-scaling pattern: gains are largest where baseline agent performance is lowest, suggesting evolved harnesses compensate for behavioral gaps weaker models cannot self-correct.
    • Most frequent AEGIS edit targets: D2 (context assembly) and D4 (tool ecosystem).

    Limitations¶

    • Gains diminish near performance ceilings; τ³-Bench yields only +1.1%, indicating limited headroom on saturated benchmarks.
    • Heterogeneous task sets require explicit variant isolation; without it, single-harness evolution stagnates or degrades.
    • The operational mirror has scope limits (§7.3): the symbolic/LLM-generative evolver can construct structured exploits that numerical RL cannot, requiring Critic safeguards that may not catch all pathologies.
    • Multi-round evolution (up to 15 rounds) carries non-trivial compute cost; cost–performance tradeoffs are discussed but not fully quantified for production settings.
    • Codebase is not yet open-sourced at time of publication.
    • Experiments use only three model families; generalization to other architectures or modalities is not demonstrated.
    • Paper text is truncated before the explicit Limitations section (§7.7), so additional stated limitations may not be captured here.

    Relevance to Agentic AI / LLM Agents¶

    HarnessX directly addresses a foundational bottleneck in agentic systems: the hand-crafted, static scaffolding that mediates how any model observes, reasons, and acts. By formalizing the harness as a first-class typed object and framing its optimization as a symbolic MDP, it provides the cleanest theoretical grounding to date for the emerging "self-evolving agent" line of work (ADAS, AFlow, SICA, Darwin Gödel Machine). The operational mirror is particularly valuable as a design tool — it converts well-understood RL failure modes into concrete architectural defenses for harness engineers. The co-evolution result (+4.7% on top of harness-only gains) demonstrates that harness optimization and model training are not independent problems but mutually reinforcing, which has direct implications for how agentic training pipelines should be architected going forward.
