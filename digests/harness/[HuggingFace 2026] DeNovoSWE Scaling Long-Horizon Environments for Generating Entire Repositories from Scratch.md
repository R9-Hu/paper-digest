# DeNovoSWE: Scaling Long-Horizon Environments for Generating Entire Repositories from Scratch

*🕒 **Published (v1):** 2026-06-09 11:37 UTC  ·  **Source:** HuggingFace  ·  [link](https://huggingface.co/papers/2606.10728)*

## TL;DR

DeNovoSWE is a large-scale dataset of 4,818 whole-repository-generation tasks, each requiring an agent to implement a complete Python repository from synthesized documentation. It is constructed via a sandboxed multi-agent divide-and-conquer pipeline with iterative critic-repair, and uses a difficulty-aware trajectory filtering strategy for SFT data curation. Fine-tuning Qwen3-30B-A3B on this data raises its BeyondSWE-Doc2Repo score from 5.8% to 47.2%.

## Problem

Existing SWE training datasets (SWE-Gym, Scale-SWE, OpenSWE) are almost entirely centered on single-issue bug fixes in existing codebases. This leaves a critical data gap for long-horizon, whole-repository generation—a qualitatively harder task requiring architect-level planning across interdependent files, APIs, and components. Benchmarks (BeyondSWE, NL2RepoBench) expose this gap but contain only 50–104 evaluation instances, far too few for training. Additionally, constructing verifiable repository-generation tasks is non-trivial: raw GitHub documentation is incomplete, evaluation must be executable, and agents can exploit leakage channels to recover reference implementations.

## Method

**Dataset construction pipeline (DeNovoSWE-Data):**

1. **Divide phase:** An overview-writer agent summarizes the repository; a capability-writer agent partitions it into functional capabilities and maps each to concrete modules/functions/classes. Repository profiling runs unit tests to capture runtime traces, classifying components as direct, core-indirect, or non-core-indirect—only the first two are documented, preserving task realism.
2. **Conquer phase (Draft-Critic-Repair):** For each capability, a draft agent generates initial documentation in a sandboxed environment with source access. A critic agent checks structural completeness and API coverage using a rule-based missing-component detector, then a repair agent addresses identified gaps. This loop repeats until the iteration budget is exhausted. All agent modules use GPT-5.4/5.5.
3. **Leakage prevention:** Pre-cleanup strips source code, erases pip/wheel caches and `/tmp` compilation artifacts, and destroys git history. Runtime enforcement blocks network-based source recovery (pip download, curl/wget targeting the package); a fuzzy-match filter and LLM-as-judge audit catch circumvention attempts.
4. **Repository selection:** Docker environments built via Scale-SWE; filtered to ≥90% original test pass rate and ≥50% test coverage.

**Difficulty-aware trajectory filtering:**
- Three difficulty signals per instance: structural (log-normalized executable line count), and two independent LLM difficulty ratings (5-level ordinal, linearly mapped to [0,1]).
- Weights fused via convex combination; optimal weights found by maximizing absolute Pearson correlation with empirical rollout pass rates on a calibration subset (grid search over the 2-simplex).
- Five difficulty tiers with monotonically relaxed score thresholds: [0.0,0.2)→0.90, [0.2,0.4)→0.85, [0.4,0.6)→0.80, [0.6,0.8)→0.70, [0.8,1.0]→0.60.
- Trajectories generated by DeepSeek-V4-Pro (3 rollouts per instance, 3 more for failures); ~11k trajectories retained after filtering. SFT uses loss masking on failed tool invocations and heredoc operations.

## Key Contributions

- **DeNovoSWE-Data**: 4,818 verifiable document-to-repository instances, ~46× larger than prior dataset NL2Repo (104 instances); average test coverage 85.5%, median 89.6%.
- **Automated sandboxed construction pipeline** combining divide-and-conquer capability decomposition, runtime trace profiling, and iterative critic-repair documentation synthesis—no human annotation.
- **Difficulty-aware trajectory filtering** with instance-specific thresholds calibrated against empirical rollout pass rates, outperforming fixed global thresholds.
- **DeNovoSWE-Agent**: fine-tuned Qwen3-30B-A3B and Qwen3.5-35B-A3B models demonstrating substantial gains on held-out benchmarks.

## Results

- Qwen3-30B-A3B-Instruct (base): 5.8% BeyondSWE-Doc2Repo, 4.3% NL2RepoBench.
- Scale-SWE-Agent (issue-level data): 29.2% Doc2Repo, 18.3% NL2Repo.
- **DeNovoSWE-Agent-30A3B**: **47.2% Doc2Repo**, **23.0% NL2Repo**.
- **DeNovoSWE-Agent-35A3B**: **50.0% Doc2Repo**, **27.1% NL2Repo** (vs. 43.8%/23.5% base).
- Within 2.0 pp of Gemini3-Pro (52.0%) and 2.4 pp of Qwen3.5-Plus (52.4%) on Doc2Repo.
- Best proprietary model (GPT-5.4 CodeX): 61.7% Doc2Repo.
- Ablation: difficulty-aware filtering (0.500) outperforms best fixed-threshold baseline (0.488) on Doc2Repo; same pattern on NL2Repo (0.271 vs. 0.264).
- All results averaged over 3 independent trials.

## Limitations

- Dataset is Python-only (via Scale-SWE Docker framework); generalization to other languages not demonstrated.
- Documentation quality depends on GPT-5.4/5.5 at construction time; errors in synthesis propagate to task validity.
- Leakage prevention is heuristic: fuzzy-match filtering and LLM-as-judge auditing may not catch all circumvention strategies.
- Difficulty scoring weights are calibrated on a subset I⋆ where rollouts were available; this introduces a circularity if the calibration subset is not representative.
- NL2RepoBench results for proprietary models are not reported (dashes in Table 3), making full comparison with frontier systems incomplete.
- SFT-only training; no RL or preference-optimization experiments are reported.

## Relevance to Harnesses / Meta-Harnesses

DeNovoSWE is itself a multi-agent construction harness: a sandboxed pipeline of specialized agents (overview writer, capability writer, profiler, draft/critic/repair) that orchestrate each other to produce training data at scale, with LLM-as-judge arbitration and strict sandboxing/containment policies enforced at the environment level. The leakage-prevention infrastructure—runtime command restriction, multi-channel cache purging, git-history sanitization, and LLM-based circumvention auditing—is a concrete engineering template for harness-level containment, directly relevant to work on agentic containment and evaluation integrity. The difficulty-aware filtering mechanism addresses a meta-level challenge for harnesses that generate their own training data: how to curate trajectories from a heterogeneous task distribution without biasing coverage toward easy instances.
