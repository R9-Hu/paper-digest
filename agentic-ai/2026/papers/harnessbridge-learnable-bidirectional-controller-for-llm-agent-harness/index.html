    HarnessBridge: Learnable Bidirectional Controller for LLM Agent Harness¶

    🕒 Published (v1): 2026-06-11 00:00 UTC · Source: HuggingFace · link

    Why this paper was selected

    HarnessBridge: learnable bidirectional controller between LLM agent and harness

    TL;DR¶

    HarnessBridge replaces manually engineered LLM agent harnesses with a lightweight learnable controller that mediates bidirectional agent–environment interaction via two learned projections: one that compresses raw trajectories into compact, decision-relevant states, and one that filters proposed actions before committing them to the environment. Trained on a curated harness supervision dataset via unified instruction tuning on Qwen3.5-0.8B, it matches or surpasses specialized harnesses on Terminal-Bench 2.0 and SWE-bench Verified while cutting token usage by up to 90% and generalizing to unseen commercial models.

    Problem¶

    Harnesses—the scaffolds that format observations, manage context, invoke tools, parse outputs, and validate actions—are almost universally hand-engineered, making them brittle and expensive to scale as interaction trajectories grow longer. In the environment-to-agent direction, trajectories accumulate stale, redundant, or low-signal context that inflates token cost and degrades decision quality. In the agent-to-environment direction, agents repeat ineffective actions, pursue invalidated hypotheses, or issue malformed commands, wasting scarce environment steps. No prior work learns the runtime interaction policy itself as an end-to-end trainable object.

    Method¶

    HarnessBridge parameterizes the harness as a learnable bidirectional policy \(\pi_h\) that transforms the raw interaction at each turn:

    \[\pi_h : (s, q, H_t, a_t) \mapsto (\tilde{H}_t, a'_t)\]

    where \(\tilde{H}_t\) is the generator-visible state and \(a'_t\) is the environment-facing action.

    Observation projection \(P_\text{obs}\) maps raw history \(H_t = (h_1, \ldots, h_t)\) to a projected view \(\tilde{H}_t = (U_t, \tilde{h}_1, \ldots, \tilde{h}_t)\). An active-state index \(U_t\) (unresolved errors, open constraints, pending goals) is prepended. Each history unit \(h_i\) receives an exposure decision \(z_i \in \{\text{Pass, Compress, Drop}\}\).

    Action projection \(P_\text{act}\) emits a pass/reject decision \(d_t\) for each proposed action \(a_t\). Rejections produce structured trajectory-grounded feedback \(\rho_t = (\text{concern}, \text{evidence}, \text{suggestion})\); if no trajectory evidence supports rejection, \(P_\text{act}\) defaults to Pass. The raw trajectory \(H_t\) is always preserved as the authoritative record—compression only affects the generator-visible view.

    Both projections are unified into a single shared policy \(P_\theta\), initialized from Qwen3.5-0.8B and fine-tuned via unified instruction tuning: each projection task is cast as instruction-following with different prompt templates and output formats. Training data is curated by prompting instruction-tuned models to produce harness traces, filtering for successful trajectories, and applying an LLM judge for projection quality.

    Key Contributions¶

    • First formulation of harness engineering as an end-to-end learnable generation problem, replacing static hand-coded interaction logic with a trainable policy.
    • A bidirectional harness controller with two components—observation projection (PASS/COMPRESS/DROP per history unit + active-state index) and action projection (PASS/REJECT with grounded feedback)—unified under a single lightweight model.
    • Unified instruction tuning over a curated harness supervision dataset covering both projection directions.
    • Demonstrated cross-generator generalization: a harness trained on Qwen3.5-35B-A3B traces transfers to GPT-5.4, DeepSeek-V4, and Claude-Opus-4.7 with preserved or improved task performance.
    • Robustness to supervision source: HarnessBridge-D trained on DeepSeek-V4 traces generalizes comparably.

    Results¶

    Terminal-Bench 2.0 (out-of-domain): - Qwen3.5-35B-A3B: SR 33.7% vs. Terminus 2 baseline 30.3% (+11.2 pp); token usage 1.23M vs. 2.31M (−46.8%) - GLM-4.7-Flash: SR 20.7% vs. 19.1% (+8.4 pp reported in text; table shows +8.4 pp); token usage reduced by −77.0% (1.23M implied)

    SWE-bench Verified: - Qwen3.5-35B-A3B: SR 60.2% vs. Terminus 2 61.6% (−2.3 pp, competitive); token 1.13M vs. 1.47M (−23.1%), lowest token budget among all baselines - GLM-4.7-Flash: SR 45.2% vs. Terminus 2 45.2% (matched); token −2.0%

    Generalization to commercial models (Terminal-Bench 2.0): - GPT-5.4-Nano: SR 22.5% vs. 18.0% (+25.0%), tokens 0.91M vs. 9.80M (−90.7%) - GPT-5.4: SR 53.9% (matched), tokens 0.99M vs. 9.41M (−89.5%) - DeepSeek-V4-Flash: SR +9.1%, tokens −44.0% - Claude-Opus-4.7: SR 65.2% vs. 64.0% (+1.9%), tokens −26.9%

    Ablations (Terminal-Bench 2.0): Removing either projection reduces success rate on both Qwen3.5-35B-A3B and GLM-4.7-Flash; some ablated variants lower token use but at consistent SR cost.

    Limitations¶

    • HarnessBridge is trained only on SWE-bench trajectories; Terminal-Bench 2.0 is used as an out-of-domain test, but broader evaluation across diverse agent domains (web navigation, multi-modal tasks) is not reported.
    • The harness controller (Qwen3.5-0.8B) adds latency per turn; wall-clock overhead is not quantified.
    • Data curation relies on LLM judges to assess projection quality, potentially inheriting judge biases.
    • Action rejection feedback is conditioned on trajectory grounding, but the failure modes of incorrect rejections (blocking valid actions) are not empirically characterized.
    • All coding benchmarks; generalizability to non-coding agentic tasks is assumed but untested.

    Relevance to Agentic AI / LLM Agents¶

    HarnessBridge directly addresses a neglected axis in agent system design—the harness—by making it a first-class learnable component rather than fixed infrastructure, which is particularly important as agents tackle longer-horizon tasks where context bloat and unproductive action cycles compound. The bidirectional framing (what the agent sees vs. what the environment executes) provides a principled abstraction that complements both stronger generators and richer environment design, fitting naturally into the emerging "auto-harness" research direction initiated by Meta-Harness. The cross-generator generalization result is notable: a 0.8B harness trained on one model family transfers to GPT-5.4 and Claude-Opus-4.7, suggesting the learned interface policy captures structural properties of effective trajectory management rather than model-specific artifacts. For practitioners building production LLM agent systems, the 89–91% token reduction on high-token generators is a direct engineering contribution.
