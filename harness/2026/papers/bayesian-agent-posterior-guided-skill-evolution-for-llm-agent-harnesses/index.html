    Bayesian-Agent: Posterior-Guided Skill Evolution for LLM Agent Harnesses¶

    🕒 Published (v1): 2026-06-06 00:00 UTC · Source: HuggingFace · link

    Why this paper was selected

    Bayesian posterior-guided skill evolution directly optimizes LLM agent harness assets

    TL;DR¶

    Bayesian-Agent replaces heuristic skill/SOP revision in LLM agent harnesses with a feature-conditioned categorical posterior over each reusable skill, mapping belief state to five auditable actions (explore, patch, split, compress, retire) without touching model weights. An incremental repair mode ingests verified trajectories from an existing baseline run and reruns only failed tasks with posterior-guided skill context. On three benchmarks with deepseek-v4-flash, incremental repair raises SOP-Bench from 80% to 95%, Lifelong AgentBench from 90% to 100%, and RealFin-Bench from 45% to 65%.

    Problem¶

    Agent harnesses accumulate skills and SOPs through LLM-generated self-reflection or by reusing observed successes and failures as if raw counts were reliable beliefs. Sparse agent trajectories are rarely i.i.d., so a single failure can be noise or the first signal of a reusable failure mode — frequency-based estimators cannot distinguish these cases and admit noisy edits that degrade future tasks. Conversely, a harness that never revises its skills leaves repeated failures outside any learning loop.

    Method¶

    The framework formalizes each reusable harness skill \(h_k\) as a hypothesis about whether a frozen model \(M_\theta\) will succeed under inference environment \(C_t = (P_t, R_t, A_t, V_t)\) (prompt, retrieved context, tool interface, verifier). The target quantity is

    \[p_{k,t} = P(y_t = 1 \mid M_\theta, C_t, h_k, z_t),\]

    where \(z_t = g(e_t)\) is a discrete feature vector extracted from a verified trajectory, bucketing token count, turn count, latency, and failure mode.

    Evidence model. A factorized categorical Naive Bayes with Laplace smoothing \(\lambda=1\) maintains class prior \(\pi_{k,t}(\ell)\) and per-feature likelihoods \(\theta_{k,j,t}^{(\ell)}(v)\) (Eqs. 7–9). The success posterior used for ranking is

    \[s_{k,t}(z) = \frac{\tilde{p}_{k,t}(1 \mid z)}{\tilde{p}_{k,t}(0 \mid z) + \tilde{p}_{k,t}(1 \mid z)}.\]

    A conjugate Beta-Bernoulli summary \((\alpha_{k,t}, \beta_{k,t})\) is retained for audit and conservative failure-dominance checks.

    Posterior-guided policy. An ordered decision rule (Eq. 12) maps belief state \(B_k\) to one of five actions: - Explore — no observations yet, or posterior uncertain - Retire — \(\beta_k \geq 4\) and \(s_k(\emptyset) < 0.45\) - Patch — same failure mode appears \(\geq 2\) times; injects a concrete guardrail into the model-facing prompt - Split — \(\geq 3\) distinct contexts, \(\geq 4\) observations; separates a broad SOP into context-specific skills - Compress — \(\geq 3\) observations and \(s_k(\emptyset) \geq 0.72\); keeps reliable skills concise to avoid crowding context

    Posterior numbers are kept in an audit layer; the model-facing prompt receives only executable skill text and failure-mode patches.

    Execution modes. Full mode: registry starts empty, updated online after every task. Incremental mode: ingests a completed baseline run's verified traces, then reruns only failed tasks with posterior-guided skill context.

    Adapter contract. The same Bayesian layer wraps external harnesses (GenericAgent, mini-swe-agent, Claude Code) via a trajectory schema boundary; Bayesian-Agent owns evidence ingestion, belief updates, policy decisions, and skill-context rendering.

    Key Contributions¶

    • Reformulation of harness skill/SOP maintenance as a posterior-inference problem over a frozen model's success probability, providing a unified language for prompt, context, and harness engineering
    • Feature-conditioned categorical evidence model with Laplace smoothing and factorized likelihood, operating on discretized trajectory signals
    • Five-action posterior-guided rewrite policy with explicit, inspectable triggers (Table 1)
    • Incremental repair mode that plugs into any existing agent run and targets only failed tasks
    • Native minimal backend plus adapter contract allowing the same Bayesian layer to operate over GenericAgent, mini-swe-agent, and Claude Code
    • Empirical study spanning three benchmarks (SOP-Bench, Lifelong AgentBench, RealFin-Bench), two DeepSeek backbones, and four execution backends, including negative and saturated cases

    Results¶

    • BA-Inc, deepseek-v4-flash: SOP-Bench 80%→95% (3/4 failures repaired); Lifelong AgentBench 90%→100% (2/2 failures repaired, 84k repair tokens); RealFin-Bench 45%→65% (8/22 failures repaired)
    • BA-Full, deepseek-v4-flash: SOP-Bench 80%→95%, RealFin-Bench 45%→52.5%, with reduced total tokens vs. GA on both; Lifelong AgentBench drops from 90% (GA) to 85% — a negative case from sparse online evidence
    • deepseek-v4-pro (partly saturated): GA already solves all SOP/Lifelong tasks; RealFin GA 60%→BA-Full 65%→BA-Inc 68% (3/16 failures repaired)
    • Native backend, flash: SOP-Bench 95%→100%, Lifelong AgentBench 95%→100%, RealFin-Bench 62.5%→72.5% (BA-Inc)
    • Claude Code backend, flash: SOP-Bench BA-Full 90%→100%; RealFin-Bench BA-Inc 77.5%→87.5%
    • Claude Code backend, pro-1m: SOP-Bench BA-Full 65%→95%; Lifelong AgentBench saturated at 100%; RealFin-Bench BA-Inc improves from 65%→75%
    • Token efficiency (accuracy/total-tokens) of BA-Inc on flash is 19.63 (SOP), 23.85 (Lifelong), 3.96 (RealFin) vs. GA's 11.47, 26.07, 4.24

    Limitations¶

    • BA-Full can degrade performance when evidence is sparse (Lifelong AgentBench flash: 90%→85%), indicating online skill evolution has ordering/cost sensitivity
    • Policy thresholds (e.g., \(\beta_k \geq 4\), \(s_k < 0.45\) for retire) are implementation defaults, not empirically optimized
    • No repeated-trial error estimates; all accuracy figures are single consolidated benchmark runs
    • Incremental-mode token counts exclude the upstream baseline run, making direct total-cost comparison with BA-Full asymmetric
    • Does not claim full Bayesian model selection over competing skill hypotheses; Beta-Bernoulli is retained for audit only
    • Some residual RealFin-Bench failures stem from missing cache paths or domain-data unavailability rather than reasoning deficits, complicating interpretation of the ceiling

    Relevance to Harnesses / Meta-Harnesses¶

    Bayesian-Agent is directly a harness-management framework: it treats the external inference environment — prompts, SOPs, skills, tools, memory — as the primary locus of agent capability rather than model weights, formalizing the harness as an evidence-bearing system that should be optimized under uncertainty. The five-action posterior-guided policy (explore/patch/split/compress/retire) provides a principled alternative to ad-hoc heuristic reflection, making harness evolution auditable and locally evidence-grounded. The adapter contract design — same Bayesian layer over multiple backends including Claude Code and mini-swe-agent — demonstrates that harness-level skill management can be decoupled from any particular runtime, a key requirement for meta-harness architectures. The incremental repair mode is directly applicable to any existing harness: ingest its verified traces, identify failures, inject posterior-guided patches, closing the loop without retraining.
